Creating a Multi-Brand Design System

Overview

Since the beginning of 2020 the main objective for Tapestry’s digital team has been to bring all of its brands onto a unified backend codebase, where brands can leverage shared feature sets, streamlined support, and simplified growth, change, and improvement. Part of this effort is to create a design system that can accommodate all of Tapestry’s brands.

A well implemented design system is meant to increase efficiency and consistency when building products at scale by utilizing reusable components and patterns that are guided by clear standards. Now, how can we achieve that with not one but 3+ unique brands? How do we rework the current system for it to be flexible enough to accommodate multiple brands and at the same time maintain efficiency by having a unified source of truth?

Motivation & User Insights

During my first few months at Tapestry, I primarily worked on building out the component library and page templates for Coach using the existing library that was built for Stuart Weitzman (the first brand that got migrated to the codebase). During that process, I noticed some inefficiencies and drawbacks with the current approach.

In theory, Coach would reuse the same base components and templates as Stuart Weitzman. However, due to major brand identity differences, we created new component sets and templates to accommodate all of Coach's styling differences.

Before looking into a new solution, I wanted to get more insights and feedback from other users of the design system. After talking to a range of developers and designers, these are the main pain points we've identified.

  • Having to build and rebuild common components for each brand is time consuming and redundant
  • Spending more time on maintenance and upkeep of separate component libraries and less time working on impactful enhancements
  • No clear central source of truth

Design Tokens as the Foundation

The solution we worked on was to create a centralized component library that could be used across all brands, by implementing a multi-tier design token setup.

The first token tier is called Core Tokens. They refer to the core brand’s UI attributes and usually point to a static value. The name refers to the value itself, such as “color-green”. Each brand has their own unique set of Core Color Tokens, which include all available color options for that brand.

The next tier is Semantic Tokens. These tokens define a specific design decision and are mapped back to the Core Tokens. They are named based on their specific usage and are referenced by components for styling.

As an example, when we define the background color of a primary button component we would use a Semantic Token named “action-bg-primary-default”. Then, we can map that token to a Core Token that is unique to each brand. That way we can accommodate unique design differences of each brand without changing a component property.

Impact & Next Steps

Setting up this multi-tiered design token system gives us a strong foundation as we continue to enhance our multi-brand design system. With this setup, our shared component library can act as the source of truth and can be used by all brands. Furthermore, we can start to focus more on enhancing our centralised libraries instead of spending time upkeeping separate libraries. This has significantly changed the way we work and increased productivity across the board.

With that said, we are still in the initial stages. Below are some of the main objectives on our roadmap that we plan to address.

Design token integration with Figma

We're currently exploring the idea of a direct integration from Figma to our Design Token repository - making Figma our main source of truth for tokens.

Onboard the rest of the brands to the system

We started the token implementation with Coach and are currently working on bringing on the rest of the brands into the new design token ecosystem.

Introduce new theming capabilities

During the first phase, we introduced the capability to theme Color, Typography, and Logos. We plan to introduce theming capability for other properties such as Border Radius, Border Width, Spacing, and Shadow.
